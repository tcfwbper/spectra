# Session — Entity

## Overview

A `Session` represents a single execution instance of a workflow. It composes [SessionMetadata](../session_metadata.md) (the persistable state: `ID`, `WorkflowName`, `Status`, `CreatedAt`, `UpdatedAt`, `CurrentState`, `SessionData`, `Error`) with the runtime-only state: the chronological event log (`EventHistory`), a read-write lock, and a termination notifier. A session is created by [SessionInitializer](../../runtime/session_initializer.md) when `spectra run` is invoked and persists until it reaches a terminal status (`"completed"` or `"failed"`).

SessionMetadata is embedded (anonymous field) in Session, making all metadata fields directly accessible on the Session instance (e.g., `session.ID`, `session.Status`) without requiring a `Metadata` accessor.

This file defines the **entity struct, its fields, the lock, and the cross-method invariants**. Per-method behavior is split across sibling files matching the planned Go file layout:

| Spec file | Go file | Methods |
|---|---|---|
| [`lifecycle.md`](./lifecycle.md) | `session/lifecycle.go` | `Run`, `Done`, `Fail` |
| [`data.md`](./data.md) | `session/data.go` | `UpdateSessionDataSafe`, `GetSessionDataSafe` |
| [`event_history.md`](./event_history.md) | `session/event_history.go` | `UpdateEventHistorySafe` |
| [`current_state.md`](./current_state.md) | `session/current_state.go` | `UpdateCurrentStateSafe` |
| [`getters.md`](./getters.md) | `session/getters.go` | `GetStatusSafe`, `GetCurrentStateSafe`, `GetErrorSafe` |

**Persistence Model**: The Session entity's in-memory state is the single source of truth for all runtime decisions. Persistent storage ([SessionMetadataStore](../../storage/session_metadata_store.md), [EventStore](../../storage/event_store.md)) is a best-effort, last-write-wins record for inspection and debugging. Runtime components must never read from persistent storage to determine behavior during active session execution. All state mutations occur in memory first; persistence failures are logged as warnings and never affect runtime correctness.

## Behavior

### Construction

A `Session` is constructed only by [SessionInitializer](../../runtime/session_initializer.md) during step 12 of its main flow. Construction is direct field assignment under no lock (the entity is not yet shared). Initial values:

**SessionMetadata fields** (embedded):

| Field | Initial Value |
|---|---|
| `ID` | Generated UUID |
| `WorkflowName` | Provided by Runtime |
| `Status` | `"initializing"` |
| `CreatedAt` | `now()` (POSIX seconds) |
| `UpdatedAt` | `now()` (same value as `CreatedAt`) |
| `CurrentState` | Workflow `EntryNode` |
| `SessionData` | `map[string]any{}` |
| `Error` | `nil` |

**Runtime-only fields**:

| Field | Initial Value |
|---|---|
| `EventHistory` | `[]Event{}` |
| `mu` | `sync.RWMutex{}` (zero value) |
| `terminationNotifier` | Provided by Runtime (capacity >= 2) |

After construction, all field access must go through the thread-safe methods listed in the table above.

### Lifecycle Summary

1. `Status == "initializing"` from construction until [`Run`](./lifecycle.md) succeeds.
2. `Status == "running"` after `Run`. Workflow events flow through [`EventProcessor`](../../runtime/event_processor.md) → [`UpdateEventHistorySafe`](./event_history.md) → [`TransitionToNode`](../../runtime/transition_to_node.md), which may invoke [`UpdateCurrentStateSafe`](./current_state.md) and ultimately [`Done`](./lifecycle.md) (on exit transitions).
3. `Status == "completed"` after `Done`, or `Status == "failed"` after [`Fail`](./lifecycle.md) is called from any non-terminal status.
4. Runtime observes the terminal-state notification on `terminationNotifier` and invokes [`SessionFinalizer`](../../runtime/session_finalizer.md), which closes/deletes the runtime socket and prints the final status.

### Threading Model

Session uses a single `sync.RWMutex` per entity. All mutating methods acquire the write lock; all `GetXxxSafe` methods acquire the read lock. Argument validation runs **before** lock acquisition. Notifications on `terminationNotifier` are sent **after** the write lock is released, so a slow receiver cannot stall other Session readers.

## Inputs

Session is not invoked directly by external inputs; its construction inputs are documented above and its method inputs are documented in the per-method spec files.

## Outputs

### Session Struct

Session embeds [SessionMetadata](../session_metadata.md) as an anonymous field, plus runtime-only fields:

**Embedded from SessionMetadata**:

| Field | Type | Constraints | Description |
|---|---|---|---|
| `ID` | string (UUID) | Generated by SessionInitializer; unique across all sessions | Unique session identifier |
| `WorkflowName` | string | Non-empty, PascalCase | Name of the workflow being executed |
| `Status` | string | Enum: `"initializing"`, `"running"`, `"completed"`, `"failed"` | Current execution status |
| `CreatedAt` | int64 (POSIX seconds) | > 0 | Timestamp at construction |
| `UpdatedAt` | int64 (POSIX seconds) | >= `CreatedAt` | Timestamp of last in-memory mutation; refreshed by every mutating method |
| `CurrentState` | string | Always a workflow-defined node name | Active node in the workflow state machine |
| `SessionData` | `map[string]any` | See [`data.md`](./data.md) for namespace conventions and ClaudeSessionID type rules | Session-scoped key-value store |
| `Error` | `*AgentError` or `*RuntimeError` (nullable) | Nullable; non-nil iff `Status == "failed"` | First fatal error (first-error-wins) |

(`Error` is implemented as a Go `error` field internally; the type-switch in [`Fail`](./lifecycle.md) constrains its dynamic type.)

**Runtime-only fields**:

| Field | Type | Constraints | Description |
|---|---|---|---|
| `EventHistory` | `[]Event` | Append-only, ordered by `EmittedAt` | Chronological log of emitted events; mutated only by [`UpdateEventHistorySafe`](./event_history.md) |
| `mu` | `sync.RWMutex` | Not serialized | Read-write lock protecting all Session state |
| `terminationNotifier` | `chan<- struct{}` | Capacity >= 2, never closed | Send-only channel for terminal status notifications |

## Invariants

These are the cross-method invariants. Per-method invariants live in the corresponding spec files.

1. **Status–State Consistency**:
   - `Status == "initializing"` ⇒ session is in SessionInitializer; `CurrentState` is the workflow `EntryNode`.
   - `Status == "running"` ⇒ session is processing events; `CurrentState` is any workflow-defined node.
   - `Status == "completed"` ⇒ `CurrentState` is the `to_node` of an exit transition; reached only via [`Done`](./lifecycle.md).
   - `Status == "failed"` ⇒ `Error != nil`; reached only via [`Fail`](./lifecycle.md).

2. **Timestamp Ordering**: `0 < CreatedAt <= UpdatedAt` at all times. Every successful in-memory mutation refreshes `UpdatedAt`.

3. **Error Correlation**: `Status == "failed"` iff `Error != nil`.

4. **State Machine Initialization**: A newly constructed session has `Status == "initializing"` and `CurrentState == workflow.EntryNode`.

5. **In-Memory Status Priority**: All status mutations occur in memory first under the write lock, then persistence is attempted. The in-memory status is authoritative for runtime behavior; persistent state may lag.

6. **Event `EmittedBy` Assignment**: [`EventProcessor`](../../runtime/event_processor.md) is responsible for stamping `Event.EmittedBy = Session.CurrentState` (read via [`GetCurrentStateSafe`](./getters.md)) and `Event.SessionID = Session.ID` before calling [`UpdateEventHistorySafe`](./event_history.md).

7. **Event Rejection during Initialization**: While `Status == "initializing"`, [`EventProcessor`](../../runtime/event_processor.md) must reject all event emissions. The Session entity itself does not reject; rejection happens at the EventProcessor boundary.

8. **Event History Immutability**: Once appended via [`UpdateEventHistorySafe`](./event_history.md), events are never removed, reordered, or modified.

9. **SessionData Namespace Discipline**: Keys with namespace prefixes (`logicSpec.*`, `<NodeName>.ClaudeSessionID`) follow the conventions documented in [`data.md`](./data.md). The Session does not enforce namespace ownership.

10. **ClaudeSessionID Type Discipline**: Keys matching `<NodeName>.ClaudeSessionID` always hold `string` values; this is enforced by [`UpdateSessionDataSafe`](./data.md) at write time.

11. **ClaudeSessionID Uniqueness**: Each workflow node may have at most one `<NodeName>.ClaudeSessionID` key in `SessionData`. (Enforced by single-key map semantics; the Session does not need additional logic.)

12. **Terminal State Finality**: Once `Status` reaches `"completed"` or `"failed"`, no lifecycle method may transition it elsewhere. [`Run`](./lifecycle.md) and [`Done`](./lifecycle.md) reject all terminal statuses via status preconditions. [`Fail`](./lifecycle.md) explicitly rejects both `"completed"` (cannot fail an already-completed workflow) and `"failed"` (first-error-wins).

13. **Thread-Safety via Read-Write Lock**: All access to `Status`, `CurrentState`, `EventHistory`, `SessionData`, `Error`, and `UpdatedAt` after construction goes through the methods listed in this index. Direct field access from outside the `session` package is disallowed.

14. **First Error Wins**: The first successful [`Fail`](./lifecycle.md) call is authoritative. Subsequent `Fail` calls are rejected with `"session already failed"`.

15. **Best-Effort Persistence**: All mutating methods log persistence failures as warnings and return `nil` (or only validation errors). The in-memory state remains authoritative.

16. **Event Validation Requirement**: [`UpdateEventHistorySafe`](./event_history.md) validates required fields (including `SessionID`) before appending; invalid events never enter memory.

17. **Termination Notifier Lifecycle**: `terminationNotifier` is created by [Runtime](../../runtime/runtime.md) with capacity >= 2 (validated by SessionInitializer). It is **never closed** by any component and is sent on at most twice per session lifetime (once by `Done` xor `Fail`, optionally once more by an early-timeout path that does not call `Fail`). See `Channel Lifecycle Convention` invariant in [`runtime.md`](../../runtime/runtime.md).

## Edge Cases

Cross-method edge cases are listed below. Per-method edge cases live in the corresponding spec files.

- **Initialization timeout**: If SessionInitializer's 30s timer fires after the Session entity has been constructed but before [`Run`](./lifecycle.md) succeeds, the timeout handler calls [`Fail`](./lifecycle.md) with a `*RuntimeError` (`Issuer="SessionInitializer"`, `Message="session initialization timeout exceeded 30 seconds"`). See [`session_initializer.md`](../../runtime/session_initializer.md) for the early-timeout path that does not call `Fail`.
- **Runtime socket loss mid-session**: [`RuntimeSocketManager`](../../storage/runtime_socket_manager.md) reports the loss via the listener error channel; [Runtime](../../runtime/runtime.md) constructs a `*RuntimeError` and calls [`Fail`](./lifecycle.md). Session transitions to `"failed"` in memory; persistence is best-effort.
- **OS signal (SIGINT/SIGTERM) during run**: Runtime stops the listener and calls [`SessionFinalizer`](../../runtime/session_finalizer.md) **without** mutating Session status. `Status` remains `"running"` (or `"initializing"`); the on-disk record reflects the last persisted snapshot.
- **Process crash mid-mutation**: In-memory state is lost. The on-disk record reflects the last successful persistence. Crash recovery is out of scope for this design.
- **Session deletion request**: Not part of this entity's API. Deletion is handled by the [`spectra clear`](../../cmd/spectra/clear.md) command and requires `Status` to be `"completed"` or `"failed"`.
